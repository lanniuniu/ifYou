# MySQL 基础
- database
CREATE DATABASE my_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

- table
‌主键（Primary Key）‌：唯一标识表中每一行数据的列，不能为空且不能重复（通常是 id）。
‌外键（Foreign Key）‌：用于建立表与表之间关联的列，指向另一张表的主键。
‌索引（Index）‌：类似于书籍的目录，用于加速数据查询速度。
```
// 创建表
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY AUTO_INCREMENT, // 自增主键
    username VARCHAR(255) NOT NULL, // 非空
    email VARCHAR(255) UNIQUE, // 唯一 可以为空
    age INT
)

// 插入数据
INSERT INTO users (username, email, age) 
VALUES ('alice', 'alice@example.com', 25);

// 查询
SELECT * FROM users WHERE age > 20;

-- 添加新列
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- 修改列类型
ALTER TABLE users MODIFY COLUMN age SMALLINT;

// 删除表
DROP TABLE users;

// 注释
COMMENT ""
```

- 索引
分类：数据结构&物理存储&逻辑约束
按逻辑约束分类：主键索引、唯一索引、普通索引、组合索引、其他高级索引
| 索引类型 | 适用场景 | 注意事项 |
| --- | --- | --- |
| ‌主键索引‌ | 几乎每张表必备 | 尽量选择短小、自增的整数类型 |
| ‌唯一索引‌ | 业务上不能重复的字段（如手机号、订单号） | 区分于主键，允许为空 |
| ‌普通/组合索引‌ | 高频查询的 WHERE、JOIN、ORDER BY 字段 | 遵循最左前缀原则，避免过多索引影响写性能 |
| ‌全文索引‌ | 大文本搜索 | 替代 LIKE %...% |
‌不要过度索引‌：每个索引都会降低写操作的速度，并占用磁盘空间。
‌区分度高优先‌：选择区分度高的列（如身份证、邮箱）建立索引，性别（只有男/女）建立索引意义不大。
‌覆盖索引优化‌：如果查询的列都在索引中，无需回表，性能极佳。

```
// 组合/复合索引
ALTER TABLE users ADD INDEX idx_name_age (username, age);
```

- 约束(Constraint)
分类：实体完整性约束、参照完整性约束、域完整性约束
```
CREATE TABLE employees (
    id INT AUTO_INCREMENT PRIMARY KEY,          -- 主键约束
    email VARCHAR(100) NOT NULL UNIQUE,         -- 非空 + 唯一约束
    department_id INT,
    salary DECIMAL(10, 2) DEFAULT 0.00,         -- 默认值约束
    age INT CHECK (age >= 18 AND age <= 65),    -- 检查约束
    CONSTRAINT fk_dept                          -- 外键约束（命名规范）
        FOREIGN KEY (department_id) 
        REFERENCES departments(id)
);
```

# SQL 查询
- WHERE
- LIKE
LIKE 是 SQL 中用于‌模糊匹配字符串‌的运算符，通常配合 WHERE 子句使用，通过通配符来搜索符合特定模式的数据。
%: 匹配任意数量的字符（包括零个字符）
_: 匹配单个字符
索引失效‌：当 LIKE 以 % 开头时（如 '%keyword'），MySQL 无法使用索引，MySQL InnoDB 引擎的普通索引，
    B+Tree 索引的最左前缀匹配原则‌。会导致全表扫描，性能极差。
优化建议‌：如果业务需要频繁做后缀模糊搜索，可考虑使用全文索引（FULLTEXT）或 Elasticsearch 等搜索引擎。
如果待搜索的字符串本身包含 % 或 _，需要使用 ESCAPE 关键字指定转义符
-- 查找包含“50%”字样的记录
SELECT * FROM products WHERE discount_info LIKE '%50\%%' ESCAPE '\';

- DISTINCT
DISTINCT 是 SQL 中用于‌去除查询结果中重复行‌的关键字。
DISTINCT 必须紧跟 SELECT 之后、字段名之前使用
在 DISTINCT 中，多个 NULL 值被视为‌相同‌，最终只保留一个 NULL。
‌先过滤再去重‌：大数据量场景下，先用 WHERE 缩小数据范围，再执行 DISTINCT，避免全表扫描。

- GROUP BY
GROUP BY 是 SQL 中用于‌将数据按指定列分组，并结合聚合函数进行统计计算‌的核心子句。
SQL 的执行顺序是：WHERE → GROUP BY → HAVING → ORDER BY。
```
SELECT Customer, SUM(OrderPrice) // 返回客户名和对应的总额
FROM Orders // 表名
GROUP BY Customer; // 相同客户合并为1组
```
HAVING和WHERE区别：
WHERE是过滤数据，HAVING是过滤分组后的聚合函数结果

- CASE WHEN
类似if else
```
-- 根据分数评定等级
SELECT
    name,
    score,
    CASE
        WHEN score >= 90 THEN '优秀'
        WHEN score >= 80 THEN '良好'
        WHEN score >= 60 THEN '及格'
        ELSE '不及格'
    END AS grade

```
用途：‌动态排序、行列转换‌、数据分类
必须 END 结尾

- UNION
UNION 可以将两个或多个 SELECT 查询的结果纵向拼接成一个结果集，但要求各查询的‌列数相同‌且对应列的‌数据类型兼容‌。
```
SELECT column1, column2 FROM table1
UNION
SELECT column1, column2 FROM table2;
```
UNION 与 UNION ALL 的关键区别
UNION： 会去重，由于需要扫描比较每一行来判断是否重复，执行速度相对较慢。
UNION ALL 不会去重。省去了去重和排序的额外开销，处理大数据时性能优势明显。

# 多表查询
- INNER JOIN
用于‌横向关联‌两个或多个表的核心操作，它只返回两个表中‌匹配条件完全一致‌的行。
```
SELECT A.name, B.order_date // A B 是别名
FROM users A
INNER JOIN orders B ON A.id = B.user_id; // ON 关键字 是连接条件
```
索引优化‌：确保 ON 子句中的关联列（如 user_id）在两张表中都建立了索引，否则会导致全表扫描（笛卡尔积），性能极差。
‌小表驱动大表‌：虽然 MySQL 优化器会自动调整，但理解连接顺序有助于编写更高效的 SQL。

- LEFT JOIN & RIGHT JOIN
LEFT JOIN（左外连接）是 SQL 中用于‌保留左表所有记录‌的关联操作。即使右表中没有匹配的行，左表的记录也会出现在结果中，右表对应的字段则显示为 NULL。
```
SELECT A.name, B.order_date
FROM users A
LEFT JOIN orders B ON A.id = B.user_id; // ON 关键字 是连接条件
```
索引至关重要‌：确保 ON 子句中的右表关联列（如 user_id）有索引，否则每次匹配都需要全表扫描右表，性能极差。
‌过滤条件位置‌：如果对右表的过滤条件写在 WHERE 中，LEFT JOIN 可能会退化为 INNER JOIN。若需保留左表所有行且过滤右表，应将条件写在 ON 子句中。
RIGHT JOIN 是 镜像写法，逻辑完全一致
永远使用 LEFT JOIN‌：即使逻辑上需要右连接，也通过交换表顺序改写为 LEFT JOIN。
‌代码可读性优先‌：团队协作中，统一使用 LEFT JOIN 能大幅降低代码审查和排错的时间成本。

- CROSS JOIN
‌交叉连接‌，它会产生两个表的‌笛卡尔积‌。
结果是两者行数的乘积
```
-- 语法一：显式使用 CROSS JOIN
SELECT A.name, B.color
FROM users A
CROSS JOIN colors B; // 不需要 ON 关键字
```
‌数据膨胀风险‌：两张 1000 行的表做 CROSS JOIN，结果集瞬间达到 100 万行。大表之间直接交叉连接极易导致数据库崩溃或内存溢出。
‌务必先过滤‌：如果确实需要交叉连接，应先用子查询或 CTE 缩小表的数据范围，再进行组合。

- Self JOIN
自连接‌并不是一种新的连接语法，而是‌同一张表自己连接自己‌的特殊用法，本质上是 INNER JOIN 或 LEFT JOIN 的一种应用场景。
```
SELECT
    e.name AS 员工姓名,
    m.name AS 上级姓名
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
```
别名必须‌：不使用别名会导致 SQL 无法区分两个实例，引发语法错误。

- 子查询
嵌套在另一个 SQL 语句（如 SELECT、INSERT、UPDATE、DELETE）内部的 SELECT 查询。它也被称为‌内部查询‌或‌嵌套查询‌。
子查询的结果通常作为外层查询的条件或数据源，用于解决‌需要多步逻辑‌的复杂查询问题。它必须用括号 () 包裹。
IN、EXISTS 和 NOT EXISTS 是子查询中常用的‌条件判断运算符‌，用于检查主查询的行是否存在于子查询的结果集中。
IN: 判断某个值是否‌等于‌子查询返回结果集中的任意一个值。
EXISTS: 只判断子查询是否‌至少返回一行‌，不关心具体值。一旦找到匹配行就立即返回 TRUE，停止扫描。
NOT EXISTS: 与 EXISTS 相反，子查询‌没有返回任何行‌时才为 TRUE。

‌能用 JOIN 替代尽量用 JOIN‌：在大多数情况下，MySQL 优化器对 JOIN 的优化远好于子查询，尤其是 IN 子查询可能被优化为 EXISTS 或直接关联。
‌避免大表关联子查询‌：大表使用关联子查询会导致外层每行都执行一次子查询，性能极差。
‌EXISTS 替代 IN‌：当子查询结果集很大时，EXISTS 通常比 IN 效率更高。
